Problem & Physics

What is FDTD?

Finite-Difference Time-Domain — a direct numerical solver for Maxwell's equations on a discrete grid, marching forward in time.

Applications: antenna design, photonic crystals, EMC/EMI analysis, radar cross-section, biomedical imaging

Our Setup: 2D TMz Mode

• PEC (metallic) cavity — waves bounce inside
• Three field components:
  Ez (out-of-plane electric)   Hx, Hy (in-plane magnetic)
• Grid: 200 × 200 cells, 400 time steps

Problem & Physics

The Yee Grid — Staggered Mesh

Update Equations

Hxn+½[i,j] = Hxn-½[i,j] − Δt/(μ₀·Δy) · (Ezn[i,j+1] − Ezn[i,j])
Hyn+½[i,j] = Hyn-½[i,j] + Δt/(μ₀·Δx) · (Ezn[i+1,j] − Ezn[i,j])
Ezn+1[i,j] = Ezn[i,j] + Δt/ε₀ · [
  (Hy[i,j] − Hy[i−1,j])/Δx
  − (Hx[i,j] − Hx[i,j−1])/Δy ]
Each Ez reads from its 4 neighboring Hx/Hy — and vice versa. No cell depends on results from the same time step → embarrassingly parallel.

Problem & Physics

Leap-Frog Time Stepping

Source Injection

Ez(center) += exp(−((t − t₀)/σ)²)
Gaussian pulse at grid center

Stability & Boundary

Δt ≤ 1/(c · √(1/Δx² + 1/Δy²))  (CFL)
PEC: Ez = 0 on walls → 180° phase-flip reflection

Problem & Physics

The FDTD Stencil — Data Dependencies

Why This is "Embarrassingly Parallel"

H update reads Ez at time n
Each Hx reads 2 Ez values (above & below)
Each Hy reads 2 Ez values (left & right)
E update reads H at time n+½
Each Ez reads from 4 neighbors:
Hx[i,j], Hx[i,j−1], Hy[i,j], Hy[i−1,j]
No write conflicts
All reads are from the previous time half-step — writes only go to the current field. Two cells never write to the same location.
200 × 200 = 40,000 cells can all update simultaneously — one CUDA thread per cell, zero synchronization needed within a kernel.

CPU Implementations

Version 1: Explicit Python Loops

for n in range(n_steps):
    # Update Hx
    for i in range(Nx):
        for j in range(Ny - 1):
            Hx[i, j] = Hx[i, j] - coeff_hx * \
                (Ez[i, j+1] - Ez[i, j])

    # Update Hy
    for i in range(Nx - 1):
        for j in range(Ny):
            Hy[i, j] = Hy[i, j] + coeff_hy * \
                (Ez[i+1, j] - Ez[i, j])

    # Update Ez (interior)
    for i in range(1, Nx - 1):
        for j in range(1, Ny - 1):
            Ez[i, j] = Ez[i, j] + coeff_ez * (
                (Hy[i, j] - Hy[i-1, j]) -
                (Hx[i, j] - Hx[i, j-1]))
Every cell processed sequentially
200×200 = 40,000 cells × 3 fields × 400 steps
= 48 million Python-interpreted operations

CPU Implementations

Version 2: NumPy Vectorized

# Update Hx — entire slice at once
Hx[:, :-1] -= coeff_hx * (Ez[:, 1:] - Ez[:, :-1])

# Update Hy — entire slice at once
Hy[:-1, :] += coeff_hy * (Ez[1:, :] - Ez[:-1, :])

# Update Ez — single vectorized operation
Ez[1:-1, 1:-1] += coeff_ez * (
    (Hy[1:-1, 1:-1] - Hy[:-2, 1:-1]) -
    (Hx[1:-1, 1:-1] - Hx[1:-1, :-2])
)
Array slicing = compiled C loops
Ez[:, 1:] and Ez[:, :-1] are contiguous memory views — the subtraction runs in a single optimized BLAS/C loop.

GPU — The Parallel Paradigm

Why GPU?

CPU — Few Powerful Cores

GPU — Thousands of Simple Cores

FDTD is "embarrassingly parallel" — each cell reads only from neighbors computed in the previous time step. No data dependency within a step → 200×200 = 40,000 independent threads.

GPU — The Parallel Paradigm

Domain Tiling — 200×200 Grid → Thread Blocks

Tiling Math

blockDim = (8, 32)
gridDim = (⌈200/8⌉, ⌈200/32⌉) = (25, 7)
25 × 7 = 175 blocks
Each block owns an 8×32 tile of the domain
256 threads per block
8 rows × 32 columns = 256 = good occupancy
44,800 total threads
175 × 256 = 44,800 ≥ 40,000 cells
(4,800 threads hit the boundary guard and exit early)
Why (8, 32) and not (16, 16)?
32 threads in the j-direction = one complete warp. Since arrays are row-major (contiguous in j), a full warp reads consecutive memory addresses → coalesced access = peak bandwidth.

GPU — The Parallel Paradigm

GPU Thread Hierarchy

Thread-to-Cell Mapping

i = blockIdx.x × blockDim.x + threadIdx.x
j = blockIdx.y × blockDim.y + threadIdx.y

Block Size: (8, 32)

32 threads in j-direction = one full warp

Coalesced memory access: Consecutive threads in a warp access consecutive memory addresses → maximum bandwidth utilization on the GPU memory bus.
Config: Block = (8, 32) = 256 threads/block
Grid = (⌈200/8⌉, ⌈200/32⌉) = (25, 7) = 175 blocks
Total: 44,800 threads (≥ 40,000 cells)

GPU — The Parallel Paradigm

Memory Coalescing — Why Block Shape Matters

Bad: (16, 16) — Strided Access

Good: (8, 32) — Coalesced Access

Coalesced access can be 10–16× faster than strided. Block shape (8,32) ensures the fast j-dimension aligns with the warp, matching the row-major memory layout of our arrays.

GPU — The Parallel Paradigm

Kernel Fusion & Optimization Strategy

Before: 4 kernel launches per step

H kernel → E kernel → Source kernel → PEC kernel

After: 2 kernel launches per step

H kernel → E + Source + PEC fused 
// Fused kernel — E update + source + PEC boundaries
if (i >= 1 && i < Nx-1 && j >= 1 && j < Ny-1) {
    Ez[idx] += coeff_ez * ((Hy[idx] - Hy[idx-Ny]) - (Hx[idx] - Hx[idx-1]));
}
if (i == src_x && j == src_y) {   // source: 1 thread out of 40,000
    Ez[idx] += pulse_val;
}
if (i == 0 || i == Nx-1 || j == 0 || j == Ny-1) {  // PEC: ~800 edge threads
    Ez[idx] = 0.0f;
}
Data stays on GPU — host ↔ device transfers only for occasional snapshots. The source injection branch hits 1 thread out of 40,000 (negligible divergence). PEC boundaries hit ~800 edge threads.

Three GPU Implementations

Version 3: NVIDIA Warp

import warp as wp

@wp.kernel
def update_H(
    Ez: wp.array2d(dtype=wp.float32),
    Hx: wp.array2d(dtype=wp.float32),
    Hy: wp.array2d(dtype=wp.float32),
    coeff_hx: wp.float32,
    coeff_hy: wp.float32,
    Nx: wp.int32, Ny: wp.int32
):
    i, j = wp.tid()
    if i < Nx and j < Ny - 1:
        Hx[i, j] = Hx[i, j] - coeff_hx * \
            (Ez[i, j + 1] - Ez[i, j])
    if i < Nx - 1 and j < Ny:
        Hy[i, j] = Hy[i, j] + coeff_hy * \
            (Ez[i + 1, j] - Ez[i, j])

# Launch: one thread per cell
wp.launch(update_H, dim=(Nx, Ny),
          inputs=[Ez, Hx, Hy, ...])

v1 → v2 Optimizations

1. Pre-cache wp.float32() wrappers
Avoid 4,000 object creations per run
2. Skip energy kernel on non-snapshot steps
2 launches instead of 3 for 95% of steps
3. Pre-compute all pulse values on GPU
Eliminate per-step CPU exp() calls
4. Pre-build input argument lists
Avoid list construction overhead each step
P100: v1 = 59 ms → v2 = 31 ms 1.90× faster

Design Decisions

Why float32 on GPU, float64 on CPU?

GPU uses float32

Hardware reason: GPUs have 2× more float32 throughput. Tesla T4 float64 is 1/32 of float32 speed.
Physics reason: FDTD's finite-difference error (~Δx² ≈ 10⁻⁶) is larger than float32 rounding (~10⁻⁷). Extra precision is wasted.

CPU uses float64

No cost on CPU: Modern x86 does float64 at the same speed as float32 via SSE/AVX. No reason not to use it.
Serves as reference: The float64 NumPy result is the "ground truth" we verify all GPU versions against.
Verification: max relative diff between GPU float32 and CPU float64 = 2.25 × 10⁻⁶ — physics results are identical.

Design Decisions

Why Two Kernels Per Step?

The Data Dependency

Step 1: H reads E at time n → produces H at n+½
Step 2: E reads H at time n+½ → produces E at n+1

If merged into one kernel, thread (i,j) updates Hx[i,j], then reads Hx[i,j-1] — but thread (i,j-1) in a different block might not have finished yet.

__syncthreads() only syncs within a block.
No cross-block sync exists inside a kernel.
The only global barrier is ending the kernel.

What CAN Be Merged

Hx + Hy → 1 kernel
Write to different arrays — no conflict
Ez + Source + PEC + Obstacle → 1 kernel
Same array, no conflicts (source=1 thread, PEC=edges, obstacle=branchless multiply)
H + E → CANNOT merge
E needs ALL H values finished first — requires kernel boundary

2 kernel launches per step is the minimum for leap-frog FDTD.

Design Decisions

Obstacle: Branchless  |  Shared Memory: Not Worth It

Branchless PEC Obstacle

// Branching (warp divergence):
if (obstacle[i,j] > 0.5)
    Ez[i,j] = 0.0;

// Branchless (no divergence):
Ez[i,j] *= (1.0 - obstacle[i,j]);
Branchless: Every thread executes the same multiply instruction. mask=0 → Ez unchanged, mask=1 → Ez zeroed. No warp divergence.

Why Not Shared Memory?
